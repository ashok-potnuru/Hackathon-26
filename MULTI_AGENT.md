# Multi-Agent Code-Fix System

## What This Does

When a user submits a requirement (bug report or feature request), the system automatically:
1. Searches the code graph to find relevant files
2. Reads those files from GitHub
3. Filters to only the relevant lines (graph-based + AI-based)
4. Generates a code fix (CoderAgent)
5. Adversarially reviews the fix (ReviewerAgent) — loops up to 3 times if rejected
6. Creates a Draft PR with the approved fix

---

## Agent Pipeline

```
User Requirement (from Zoho ticket)
      │
      ▼
[Stage 3: research.py]
      │
      ├── 1. PlannerAgent  (AI)
      │        ├── LLM extracts keywords from issue title + description
      │        ├── GraphNavigator.search_nodes(keywords)
      │        │       reads local tv2z_codebase_graph/graph.json
      │        │       → finds matching nodes by keyword substring
      │        ├── GraphNavigator.get_related_files()
      │        │       BFS through edges → expands to related files (max 15)
      │        └── Returns: PlanResult(target_files, keywords_extracted)
      │
      ├── 2. GitHub fetch
      │        vc.get_file(repo, path, branch) for each target file
      │        → full file content fetched from GitHub
      │
      ├── 3. GraphNavigator.get_relevant_lines()  (deterministic filter)
      │        uses source_location (line numbers) from graph nodes
      │        → extracts only lines around matched functions/classes
      │        → merges overlapping ranges, adds line number prefixes
      │
      └── Returns: code_context + plan
      │
      ▼
[Stage 4: fix_generator.py]
      │
      └── MultiAgentOrchestrator.run()
            │
            ├── 4. ExplorerAgent  (AI, strict read-only)
            │        reads the graph-filtered code sections
            │        → decides which files MUST change vs. context only
            │        → further narrows exact relevant line ranges
            │        → labels: must_change / context_only
            │        Cannot suggest fixes — read-only mode enforced in system prompt
            │
            ├── 5. CoderAgent  (AI)
            │        receives must_change + context_only files (labelled)
            │        → generates complete new file contents
            │        → returns confidence score (0.0–1.0)
            │
            ├── 6. ReviewerAgent  (AI, adversarial)
            │        runs 6 structured checks on the proposed changes:
            │          ✓ Correctness   ✓ Security      ✓ Regression risk
            │          ✓ Boundary values  ✓ Error handling  ✓ Concurrency
            │        → VERDICT: PASS  → done
            │        → VERDICT: FAIL / PARTIAL → feedback to CoderAgent (retry)
            │        → security_ok=False → hard stop (SecurityScanError)
            │        max 3 attempts total
            │
            └── Returns: FixModel
      │
      ▼
[Stage 5: pr_creator.py]  ← unchanged, creates Draft PR on GitHub
```

## Where graph.json Comes From & How It's Loaded

graph.json is generated by running `/graphify` on the TV2Z (hackathon_wlb_api) codebase.
It is **committed directly into this repo** at `tv2z_codebase_graph/graph.json` and read from local disk.

```
Step 1: Run /graphify on the TV2Z codebase
Step 2: Copy graphify-out/ → tv2z_codebase_graph/ in this repo
Step 3: get_navigator() reads graph.json from disk once, caches in memory for the process
```

When the target codebase changes significantly → re-run `/graphify` → replace `tv2z_codebase_graph/graph.json`.

graph.json contains:
- **2,211 nodes** — files, classes, functions, each with `source_location` (line number)
- **3,142 edges** — calls, references, contains relationships
- **386 communities** — logical clusters of related code

## How graph.json Filters File Contents (Smart Extraction)

Instead of blindly sending full files, we extract only relevant lines using graph node line numbers.

```
Issue: "Fix payment timeout for international orders"
         │
         ▼
PlannerAgent → keywords: ["payment", "timeout", "international"]
         │
         ▼
GraphNavigator.search_nodes(keywords)
→ finds node: charge() at L42 in payments_controller.js  (score: 3)
         │
         ▼
GraphNavigator.get_related_files(seeds, max_hops=2, max_files=15)
→ BFS via calls/references edges
→ [payments_controller.js, order_service.js, gateway_client.js, ...]
         │
         ▼
GitHub API → fetch FULL file content for each path
         │
         ▼
GraphNavigator.get_relevant_lines(file_path, full_content, keywords)
→ finds nodes matching keywords → their line numbers (e.g. L42)
→ extracts lines 32–82 (L42 ± 40 context lines)
→ merges overlapping ranges
→ sends ONLY those lines to ExplorerAgent/CoderAgent (with line numbers prepended)

32| function processPayment(card, amount, country) {
...
42|   const result = await charge(card, amount);   ← matched node
...
```

**Fallback:** if no graph nodes match keywords for a file → sends first 3000 chars.

## ReviewerAgent — Adversarial Checks

Inspired by openclaude's VerificationAgent. The reviewer does NOT confirm the happy path —
it tries to find what's wrong. Every review runs all 6 checks:

| Check | What it looks for |
|---|---|
| Correctness | Does the fix address the root cause, not just the symptom? |
| Security | SQL injection, XSS, auth bypass, unvalidated input, data exposure |
| Regression risk | Callers of changed functions that could silently break |
| Boundary values | null, undefined, empty string, 0, negative numbers, MAX_INT |
| Error handling | Errors caught and handled, not silently swallowed |
| Concurrency | Parallel calls — state corruption or deadlock risk |

**VERDICT format** (from openclaude pattern):
- `VERDICT: PASS` → fix approved, proceed to PR
- `VERDICT: FAIL` → specific feedback sent back to CoderAgent (retry)
- `VERDICT: PARTIAL` → also triggers retry (treated same as FAIL)
- `security_ok: false` → hard stop, `SecurityScanError` raised immediately

## Agent Loop Detail

```
Attempt 1:
  ExplorerAgent → labels files: must_change=[payments.js], context=[order.js]
  CoderAgent → generates fix (confidence=0.72)
  ReviewerAgent →
    correctness: PASS
    security: PASS
    boundary_values: FAIL — charge() does not handle amount=0 or negative
    VERDICT: FAIL
    feedback: "Add validation: reject amount <= 0 before calling gateway"
  → feedback passed back to CoderAgent →

Attempt 2:
  CoderAgent → generates fix with feedback (confidence=0.88)
  ReviewerAgent →
    all checks: PASS
    VERDICT: PASS ✓
  → FixModel returned → PR created

Max 3 attempts. If never PASS → FixGenerationError raised.
If security_ok=False → SecurityScanError raised immediately (no retry).
```

## What Was Taken from openclaude

| openclaude pattern | Applied where |
|---|---|
| READ-ONLY MODE strict prohibition in system prompt | ExplorerAgent, PlannerAgent |
| Adversarial mindset ("find the last 20%") | ReviewerAgent system prompt |
| 6-check structured review | ReviewerAgent prompt + `ReviewResult.checks` dict |
| VERDICT: PASS / FAIL / PARTIAL format | ReviewerAgent — PARTIAL triggers retry like FAIL |

**Not taken** (not applicable): running shell commands, browser automation, coordinator message-passing.
